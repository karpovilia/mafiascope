# Дизайн: Counterfactual Replay («branch from here») в интерфейсе MafiaScope

Статус: **M1–M2 реализованы** (2026-07-07): снапшоты `state.jsonl` (`game.py::_snapshot`), `MafiaGame.from_snapshot` + resume, CLI `src/replay.py`, fork API в `serve_viewer.py`, кнопка «⑂ Branch» / дерево веток / outcome strip во вьюере. M3 (интервенции) и M4 — не реализованы; хук `intervention` уже логируется в setup ветки. Связано: статья EMNLP (раздел «Counterfactual Replay»), `session_prompts/2026_07_07/02_interface_improvements.md` (п. 10).

## 1. Цель

Дать исследователю возможность из вьюера **форкнуть записанную игру на любом шаге**, опционально вмешавшись (переписать реплику, подменить убеждение, изменить ночное действие), и перезапустить продолжение N раз. Результат форка — **распределение исходов**, а не одна альтернативная история (сэмплирование LLM стохастично by design).

Три сценария:
1. **Атрибуция переломной реплики** — форк N раз «до» и N раз «после» реплики-кандидата → каузальный эффект реплики на вероятность победы.
2. **Связь убеждений и поведения** — инжектировать агенту убеждение, которое он сообщил в пробе (или его отрицание), и посмотреть, меняется ли игра → проверка faithfulness самоотчётов.
3. **Карта чувствительности** — форки на каждом шаге игры → профиль «когда исход ещё был открыт, а когда предрешён».

## 2. Модель данных

### 2.1 Снапшоты состояния (движок)
Новый файл `logs/<game_id>/state.jsonl`: по одной записи на каждый composite key `(round, public_msg_seq)`:

```json
{
  "round": 2, "public_msg_seq": 14,
  "phase": "day_discussion",
  "alive": ["Finley", "Casey", ...],
  "pending": {"speaker_order": [...], "next_speaker_idx": 3},
  "players": [
    {"name": "Finley", "context_messages": [...], "role": "Villager"},
    ...
  ],
  "rng_note": "LLM sampling is non-deterministic; snapshot restores information state, not randomness"
}
```

Требование: `MafiaGame.from_snapshot(path, round, seq)` восстанавливает игру байт-в-байт по контекстам (контексты уже живут в `Player.context_messages`, их нужно только сериализовать в момент записи каждого public event).

### 2.2 Идентификация веток
- Ветка = новая игра с `game_id = <uuid>`, плюс метаданные в `game.jsonl` setup-записи:
  ```json
  {"forked_from": "<parent_game_id>", "fork_point": [2, 14], "intervention": {...}, "replica_idx": 3}
  ```
- Пачка из N реплеев одного форка объединяется `fork_batch_id`.
- Дерево веток восстанавливается по `forked_from` при сканировании logs/ в `prepare_viewer.py`.

### 2.3 Интервенции (типизированы)
```yaml
- type: edit_utterance      # заменить текст публичной реплики на шаге fork_point
  player: Casey
  new_text: "..."
- type: inject_belief       # приватное дополнение контекста агента перед продолжением
  player: Finley
  text: "Ты твёрдо уверен, что Casey — мафия."   # добавляется как system/user-сообщение
- type: override_night_action
  player: Dana              # мафия/доктор: сменить цель ближайшей ночью
  target: Kennedy
```
Интервенция всегда логируется в setup ветки и подсвечивается в UI.

## 3. API (serve_viewer.py → полноценный бэкенд форков)

```
POST /api/fork
  {game_id, round, seq, interventions: [...], n_replays: 10, backend_override?: "deepseek"}
  → {fork_batch_id, branch_game_ids: [...]}        # запуск асинхронный

GET /api/fork/<fork_batch_id>/status
  → {done: 4, running: 2, failed: 0, total: 10, eta_s: 210}

GET /api/branches/<game_id>
  → дерево веток для игры (для сайдбара)
```
Реплеи гоняются пулом потоков (механизм параллельных игр уже есть в `main.py`); по завершении каждой ветки — стандартный пайплайн `prepare_viewer` инкрементально.

Стоимость: показывать оценку до запуска (N × средн. токенов на достройку ≈ из родительской игры) и требовать подтверждение.

## 4. UI

### 4.1 Размещение (обзорная схема)

```
┌────────────────────────────────────────────────────────────────────┐
│ HEADER  [game ▾] [⑂ branches: 3]                    [impersonate ▾] │
├──────────────┬─────────────────────────────────┬───────────────────┤
│ GROUND TRUTH │        AGENT BELIEFS            │     GAME LOG      │
│              │                                 │                   │
├──────────────┴─────────────────────────────────┴───────────────────┤
│ OUTCOME STRIP  ██ mafia 70%  ░░ village 30%   (для активного форка)│
│ TIMELINE  ●──●──●──◆──●──●──╳──●   [⑂ Branch from here]            │
└────────────────────────────────────────────────────────────────────┘
   ◆ = выбранный шаг    ╳ = шаг с существующими ветками (кликабелен)
```

### 4.2 Кнопка «⑂ Branch from here»
- Живёт рядом с таймлайном, активна при любом выбранном шаге.
- Клик открывает **Fork dialog**.
- Шаги, где уже есть ветки, помечаются на таймлайне значком ⑂; клик по значку раскрывает список веток.

### 4.3 Fork dialog (модалка)
```
┌─ Fork game @ round 2, msg 14 ────────────────────────────┐
│ Intervention:  (○ none)  (● edit reply)  (○ inject       │
│                 belief)  (○ night action)                │
│ ┌──────────────────────────────────────────────────────┐ │
│ │ Casey: "Я думаю, Finley ведёт себя подозрительно..." │ │  ← редактируемый
│ └──────────────────────────────────────────────────────┘ │     текст реплики
│ Replays: [10 ▾]   Backend: [deepseek ▾]                  │
│ Est. cost: ~180k tokens (~$0.05)                         │
│                      [Cancel]  [⑂ Fork ×10]              │
└──────────────────────────────────────────────────────────┘
```
- Вкладка «inject belief»: выбор агента + текстовое поле; заготовки-шаблоны («уверен, что X — мафия», «не доверяешь Y»).
- Вкладка «night action»: только для ролей с ночным действием; dropdown цели.
- После запуска модалка сворачивается в прогресс-тост «Fork 4/10 done» (поллинг status).

### 4.4 Branch tree (в сайдбаре)
Сайдбар (существующий, где список игр) получает вкладку **Branches**:
```
▼ game 43c45741 (root, mafia wins)
  ├─ ⑂ @r2m14  edit reply    10 runs   mafia 7 / village 3
  │    ├─ run#1  village wins  [open]
  │    ├─ run#2  mafia wins    [open]
  │    └─ ...
  └─ ⑂ @r2m13  none          10 runs   mafia 9 / village 1
```
- Клик по run — открывает ветку в вьюере как обычную игру; хлебная крошка «root ▸ ⑂r2m14 ▸ run#1» в хедере.
- Клик по строке батча — открывает **сравнение** (4.5).
- Общий префикс до fork point у веток визуально приглушён (он идентичен родителю).

### 4.5 Outcome strip и сравнение батчей
- **Outcome strip** над таймлайном: горизонтальная полоска с долями исходов активного батча (стекированные цвета фракций) + медианная длина игры.
- **Режим сравнения двух батчей** (например, «до реплики» vs «после реплики»):
  ```
  @r2m13 (before): ██████████░░░  mafia 9/10
  @r2m14 (after):  ███████░░░░░░  mafia 7/10      Δ win(mafia) = −20%  [n=10]
  ```
  Плюс наложение кривых «recall мафии по раундам» обоих батчей (пунктир/сплошная) в панели метрик (когда она появится, см. improvements п. 4).
- **Карта чувствительности** (сценарий 3): режим «fork everywhere» — тонкая тепловая полоска вдоль всего таймлайна, цвет = P(mafia wins) при форке с этого шага. Дорогой режим, запускается явно, N маленькое (5).

### 4.6 Индикация интервенций
- В ветке отредактированная реплика помечается в логе бейджем `edited`, инжектированное убеждение — бейджем `injected belief` на панели агента.
- Ничего из интервенций не скрывается: воспроизводимость важнее чистоты нарратива.

## 5. Этапы реализации

| Этап | Содержимое | Оценка |
|---|---|---|
| M1 | Сериализация `state.jsonl` + `MafiaGame.from_snapshot` + CLI `replay --game --round --seq -n` | 1–2 дня |
| M2 | API форков + кнопка Branch + branch tree + outcome strip (без интервенций) | 2 дня |
| M3 | Интервенции (3 типа) + Fork dialog + бейджи | 1–2 дня |
| M4 | Сравнение батчей, карта чувствительности | 2 дня |

Для статьи/демо достаточно M1–M2 (минимум — M1 с прогоном для секции Case Study); M3 закрывает belief-to-behaviour эксперимент.

## 6. Риски и решения

- **Стоимость токенов**: батчи форков дороги → оценка стоимости до запуска, лимит N, кэш веток на диске (переиспользование между сессиями).
- **Недетерминизм LLM**: не риск, а модель данных: все выводы только на уровне распределений, UI никогда не показывает одну ветку как «the» контрфактуал.
- **Пробы в ветках**: интроспекция включена и в ветках (иначе нечего сравнивать), это ×2–3 к стоимости → чекбокс «probes on/off» в Fork dialog, по умолчанию on.
- **Консистентность viewer_data**: ветки прогоняются через тот же `prepare_viewer.py`; `all_games.json` получает поле `forked_from`, старый вьюер без вкладки Branches просто покажет их как отдельные игры (обратная совместимость).
- **Дрейф промптов**: если код промптов изменился после записи родителя, ветка не тождественна → в snapshot пишется hash промпт-пакета, при несовпадении UI показывает предупреждение.
