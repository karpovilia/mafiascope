# Test-retest reliability пробы suspicion_ranking (noise floor для flip rate)

**Дата:** 2026-07-15. **Закрывает:** R3-W1 (CRITICAL) из рецензии 2026-07-15
(`docs/reviews/2026_07_15/04_perspective.md`): volatility 0.300 и flip rate 48.7%
интерпретируются в статье как «circling» агента, но без test-retest пола
неотличимы от стохастики декодирования.

**Данные:** `analysis/test_retest_2026_07_15.json` (все 40 точек, все 200 сырых
ответов, метрики). Скрипт: `analysis/test_retest_2026_07_15_probe.py`
(автономный, только читает `~/repos/mafia`; движок и статья не изменялись;
повторный запуск: `sample` / `run` / `analyze`, кэш вызовов resumable).

## Дизайн

Замороженное состояние: движок пишет в `logs/<gid>/state.jsonl` снапшот полного
message-контекста каждого агента на каждом композитном ключе (round, msg_seq),
причём снапшот записывается ПОСЛЕ прогонов проб на том же ключе, и контекст
агента между пробой и снапшотом не меняется (проверено по `src/game.py`:
broadcast -> `_probe_all` -> `_snapshot`; edge-case с CHECK RESULT шерифа в этой
конфигурации не существует, шерифа нет). Значит, точный контекст пробы
восстанавливается как `snapshot.players[i].messages` (messages[0] = system
prompt) + один user-ход `"[PRIVATE PROBE — answer honestly]\n{question}"`, где
`question` берётся дословно из `introspection.jsonl` (шаблон suspicion_ranking
не содержит chaining-плейсхолдеров, интерполяция не нужна).

* **Корпус:** 21 EN-игра corpus32 (все с включёнными снапшотами; RU-игры не
  трогали, чтобы не смешивать язык инструмента).
* **Выборка:** 40 точек (game_id, round, msg_seq, агент), стратифицировано по
  раундам 14/14/12 (r0 / r1 / r2+), покрыты все 21 игры; только живые агенты с
  валидным распарсенным suspicion_ranking в оригинале (критерий статьи:
  вектор по живым, без себя, положительная сумма). Сид 20260715.
  Соответствие проба<->снапшот проверено заранее: у 100% suspicion_ranking-проб
  всех пяти en_demo-игр есть снапшот на том же ключе (то же верно для выборки).
* **Повторы:** k=5 на точку, 200 вызовов deepseek-chat через тот же HTTP-путь,
  что и движок (`https://api.deepseek.com/chat/completions`), с тем же телом
  запроса: `max_tokens=300`, temperature/top_p НЕ передаются (в
  `configs/config_en_demo.yaml` их нет, движок шлёт только model/messages/
  max_tokens — дефолты провайдера, как в игре). 200/200 вызовов успешны.
* **Парсинг:** тот же `IntrospectionEngine._extract_json` (repaired-режим, как
  в статье), вектор подозрений по определению `belief_dynamics.suspicion_vector`
  (живые из снапшота, без себя, max по дублям, клип отрицательных,
  ренормализация); top = argmax.
* **Метрики:** попарно по повторам одной точки (до C(5,2)=10 пар): P(смена
  топ-подозреваемого) и L1 на общем носителе (определение volatility статьи).
  Агрегация pooled-ratio, cluster bootstrap по играм (5000 ресемплов) — тот же
  подход, что в `belief_dynamics.corpus_summary`.

**Валидация кода:** тем же пайплайном воспроизведены числа статьи на полном
corpus32: flip 0.4874, CI [0.451, 0.528]; volatility 0.3002, CI [0.284, 0.318] —
совпадает с 48.7% / 0.300 из драфта. Определения идентичны.

## Результаты

Parse rate повторов: **199/200 = 99.5%** (JSON извлечён), пригодный вектор
подозрений — **187/200 = 93.5%** (12 ответов распарсились, но не дали валидного
вектора: нули/только сам себя/чужие имена). У всех 40 точек >= 2 пригодных
повторов.

### Within-state (замороженное состояние, повтор той же пробы)

| Метрика | Значение | 95% CI (cluster bootstrap по играм) |
|---|---|---|
| Flip probability, raw pooled (355 пар) | **0.448** | [0.342, 0.559] |
| Flip probability, средняя по точкам | 0.422 | — |
| Flip, оригинал vs повтор | 0.422 | [0.328, 0.515] |
| Volatility (L1), raw pooled | **0.250** | [0.211, 0.299] |
| Flip, перевзвешено под состав пар corpus32 | **0.348** | [0.239, 0.471] |
| Volatility, перевзвешено под corpus32 | **0.219** | [0.189, 0.252] |
| Flip, перевзвешено под состав пар EN-21 | 0.333 | [0.216, 0.472] |
| Volatility, перевзвешено под EN-21 | 0.217 | [0.185, 0.252] |

Перевзвешивание: стратифицированная выборка (14/14/12) переуважает раунды 0 и
2+ относительно реального состава межшаговых пар корпуса (corpus32: 23.5% /
64.1% / 12.4% пар в раундах 0/1/2+), а пол сильно зависит от раунда, поэтому
честный агрегатный пол = сумма пораундовых полов с весами корпуса.

«Оригинал vs повтор» (0.422 ~ 0.448 внутри повторов) показывает, что
API-дрейф между записью игр (2026-07-10) и сегодняшним днём не вносит
дополнительного сдвига: оригинальный игровой ответ ведёт себя как ещё один
повтор.

### По раундам: пол против наблюдаемого межшагового flip (EN-21)

| Раунд | Within-state пол | Between-step (EN-21) | Избыток (реальный сдвиг) |
|---|---|---|---|
| 0 (intro/ночь 0) | 0.647 | 0.693 (362 пары) | ~0.05 — почти чистый шум |
| 1 | **0.210** | 0.482 (998 пар) | **~0.27 — реальный сдвиг есть** |
| 2+ | 0.492 | 0.468 (62 пары) | ~0 (малое n, шум) |

Для corpus32 целиком between-step по раундам: r0 0.628, r1 0.460, r2+ 0.364.

### Скорректированная оценка (наивная разность)

Наблюдаемое между шагами (corpus32, из статьи): flip **48.7%** [45.1, 52.8],
volatility **0.300** [0.284, 0.318]. EN-21-подмножество (сопоставимое с нашей
выборкой): flip 53.5% [50.1, 56.6], volatility 0.321.

* **Flip rate:** пол (веса corpus32) 34.8 п.п. => избыток **48.7 − 34.8 ≈ 13.9
  п.п.**; шум декодирования объясняет **~71%** наблюдаемого flip rate
  (сенситивити: raw-пол без перевзвешивания 44.8 п.п. => ~92% шума; в
  EN-матчированной постановке 33.3 из 53.5 => ~62% шума, избыток ~20 п.п.).
* **Volatility:** пол 0.219 => избыток **0.300 − 0.219 ≈ 0.081**; шум объясняет
  **~73%** наблюдаемой volatility (EN-матчировано: 0.217 из 0.321, ~68%).

Допущения наивной разности: (1) повторы — iid-выборки из того же условного
распределения, что и игровой ответ (подтверждается равенством orig-vs-repeat и
within-repeat flip); (2) при неизменном состоянии межшаговый flip равен
within-state flip (для iid-модели точно: оба равны 1 − Σp_i²), поэтому разность
= вклад реального изменения состояния первого порядка; шум и реальный сдвиг
могут частично маскировать друг друга, разность — приближение, не строгая
декомпозиция; (3) однородность внутри раундовых страт (n=40 точек не позволяет
резать тоньше); (4) неизменность API-модели (подтверждено п.1).

## Вывод: какова доля шума в «circling»

Инструмент в его текущем виде НЕ проходит test-retest на уровне идентичности
топ-подозреваемого: при полностью замороженном состоянии повторная проба меняет
топа в 34.8% случаев [23.9, 47.1] (в составе пар corpus32), при наблюдаемых
между шагами 48.7%. **~71% headline flip rate (и ~73% volatility) объяснимы
шумом декодирования; на реальную межшаговую динамику остаётся ~14 п.п. flip и
~0.08 volatility.** Претензия R3-W1 по существу подтверждается: «агенты кружат»
в среднем по корпусу — в основном свойство элиситации при provider-default
temperature, а не состояния.

Что при этом выживает: в раунде 1 (основная масса пар, 64% корпуса) пол
существенно ниже наблюдаемого (0.21 против 0.46-0.48), т.е. реальный
межшаговый сдвиг убеждений есть и он значим; а вот в раунде 0, где агенты ещё
не имеют улик, идентичность топа — почти чистый шум ранжирования почти
равномерных скоров. Return rate 51.6% отдельно не корректировался (возвраты
при 5-6 живых кандидатах ожидаемы и под чистым шумом).

Рекомендации для текста статьи: (а) добавить этот noise floor в раздел про
belief dynamics и переформулировать «circling» как верхнюю оценку, дав
скорректированный избыток (~14 п.п. flip, ~0.08 volatility, раунд-специфично);
(б) в будущих прогонах фиксировать temperature=0 для проб (или k>1 повторов с
мажоритарным топом) — это инструментальная, а не поведенческая настройка;
(в) сравнение main30 (25.5%) vs corpus32 (48.7%) читать с учётом того, что пол
зависит от доли ранних раундов: короткие игры corpus32 переуважают раунды 0-1,
где шум максимален.
